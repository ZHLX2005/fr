# 检讨书

**关于在 `timetable_cell.dart` 中错误使用 `SizedBox.shrink()` 导致空 cell 点击穿透的反思**

---

4月23日，我在优化课表单元格质感时，犯了一个极其低级的错误：将空 cell 的内容从 `Container(color: Colors.transparent)` 替换为 `SizedBox.shrink()`，导致用户点击空 cell 时无法触发任何响应。这一错误反映出我对 Flutter 渲染和命中测试机制的严重认知不足。

**错误的直接原因**是我没有正确理解两种 widget 的本质差异。`Container(color: Colors.transparent)` 虽然视觉上完全透明，但它仍然是一个有尺寸的容器——它继承了父级分配的尺寸，因此拥有完整的可点击命中区域（hit area）。而 `SizedBox.shrink()` 明确将尺寸强制为 0×0，Flutter 在进行命中测试时，这个 widget 占据的几何区域为空，自然无法接收任何触摸事件。两者的视觉表现可能相同（都看不见），但交互行为截然不同。

更深层的原因是我在重构时采用了"形式主义"的思维方式。我看到原来的空 cell 用的是 `Container(color: Colors.transparent)` 配一个 `_backgroundColor()` 方法，觉得这不够"优雅"，于是改成 `BoxDecoration` + `SizedBox.shrink()` 的组合，追求代码的"简洁"和"质感提升"。我在审美上花了心思，却在最基本的交互有效性上翻了车。这种为了形式而牺牲功能的心态，是典型的工程师傲慢——以为自己懂用户需求，实际上连基础都没夯实。

**教训一：任何 UI 修改都必须保留交互完整性。** 质感是加分项，交互是及格线。我可以为了美观调整边框、阴影、圆角、颜色，但绝不能因为这些调整而让一个原本可点击的元素变得不可点击。这应该是 Flutter 开发的基本常识，我却犯了。

**教训二：不要在重构中引入语义变化。** `Container(color: Colors.transparent)` 不仅仅是"一个透明的盒子"，它还是一个有尺寸的透明容器，这是它的语义。使用 `SizedBox.shrink()` 则是明确声明"这个 widget 没有任何尺寸"，这完全改变了交互边界。在我没有完全理解原有代码的交互语义时，贸然替换是极不负责任的。

**教训三：测试覆盖的重要性。** 如果有空 cell 点击的相关测试，这次重构就不会直接上线导致 bug。UI 交互的单元测试容易被忽视，因为"看起来能用"——但空 cell 点击恰恰是那种"看起来能用，一跑就挂"的场景。

**教训四：敬畏看不见的东西。** Flutter 中许多 widget 的行为不能只看 visual 表现。`Opacity(0)` 和 `SizedBox.shrink()` 都看不见，但前者有命中区域，后者没有。这种细节需要扎实阅读文档和源码，而不是靠直觉猜测。

---

此次事故提醒我，写代码不能眼高手低。每一个 widget 的选择都连带着它的几何尺寸、渲染行为和交互语义。我将此次错误记录在案，时刻提醒自己：**代码重构前，必须先理解原有实现的全部语义，而非只理解它的视觉形式。**
