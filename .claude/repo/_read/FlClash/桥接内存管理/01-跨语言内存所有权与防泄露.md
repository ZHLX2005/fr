# IPC/ABI/FFI 跨语言内存所有权与防泄露机制——以 FlClash 为例

> "跨语言"的最大风险不是协议错，而是**内存分配在语言 A，却由语言 B 来释放**。每一种桥接类型（IPC / ABI / FFI）都有自己的归属约定。
> 本文直接对照 FlClash 在 4 类桥接中各自怎么管、怎么防。

---

## 一、内存模型的"四个世界"

| 世界 | 内存由谁管 | 如何分配 | 如何回收 | 跨语言时如何处理 |
|---|---|---|---|---|
| **Dart (Flutter Engine VM)** | GC (generational) | Dart 对象 / boxed | GC sweep | 必须以 handle / Pointer 形式交出去，不能交原始对象 |
| **C (cgo runtime)** | 程序员手管（C 库） | `malloc / calloc / C.CString` | `free / C.free` | 必须成对，跨边界的一方拥有分配责任 |
| **Go (runtime)** | GC + arena | `make / new / unsafe.Pointer` | GC / `runtime.KeepAlive` | 必须 ensure pointer 不会被 GC 收回 |
| **Rust** | 所有权/借用 | `Box::new`, `Vec<T>`, `&T` | scope end / Drop | FFI 边界用 `Box::into_raw` + `Box::from_raw` 配对 |

四个世界的"垃圾回收时机"都不一样 —— 所以跨语言时**必须显式约定谁分配谁释放**。

## 二、第 1 类：Android Go↔Dart（CGO + 函数指针表）—— 双重所有权的核心难点

### 2.1 一边分配、另一边释放的危险

`core/lib.go:184-199` 里 `invokeAction(callback, paramsChar)` 出现两个跨语言参数：

| 参数 | 谁分配 | 谁释放 |
|---|---|---|
| `paramsChar *C.char` | Dart 侧（C.CString 复制 Dart string） | **C 侧**（在 Go 端 `takeCString` 中 C.free） |
| `callback unsafe.Pointer` | Dart 侧（`Pointer<Void>` 钉住的函数） | **Dart GC + Dart 端显式 close()** |

### 2.2 字符串分配（"谁拥有那块字节"）

`core/bride.go:32-36`：

```go
func takeCString(s *C.char) string {
    defer C.free_string(s)        // 立刻释放掉 C 的字符串
    return C.GoString(s)          // 把数据 *复制* 进 Go 的 string（自己 new 字节）
}
```

注意 `C.GoString` —— 它把 C 字符串**内容按值复制**到 Go 的 string，**之后 Go 端与那块 C 内存完全无关**。这是一条关键原则：**桥函数拿到任何 C string 立刻复制并释放原指针**，避免一份内存在两 runtime 间来回传递。

### 2.3 反向：Go 端 → Dart 端的字符串传出

```go
// core/bride.go:23-27
func invokeResult(callback unsafe.Pointer, data string) {
    s := C.CString(data)         // 在 C heap 上分配
    defer C.free(unsafe.Pointer(s))   // 立刻释放？
    C.result(callback, s)        // 跳到 Dart
}
```

⚠️ **表面看是 defer free，似乎内存立即回收。但 C.result 内部是 Dart 端拿这个指针之后立即调用 `cast<Utf8>().toDartString()` 把内容拷回 Dart string —— Dart 端一旦拷贝，原始指针立刻游离。**

所以这一段是**安全的"伪同步"释放**：Go 端 free 必须发生在 Dart 端拷贝完成之后。证据是 Dart 端调用 `C.result(callback, s)` 时是**同步 inline 调用**，未返回前不会让 Go 释放。

### 2.4 回调指针的所有权 — 最容易泄露的点

`core/lib.go:60-70`：

```go
func (th *TunHandler) clear() {
    th.removeHook()
    if th.listener != nil {
        _ = th.listener.Close()
    }
    if th.callback != nil {
        releaseObject(th.callback)
    }
    th.callback = nil
    th.listener = nil
}
```

`th.callback` 是 Dart 端传过来的 function pointer。在 TUN 整个生命周期里**一直被保留**，防止 Go GC/编译把它回收掉。最终在 `clear()` 里通过 `releaseObject` 把所有权归还给 Dart。

⚠️ 注意：只有当 `result.Method != messageMethod` 时才 release（`core/lib.go:162-171`）：

```go
func (result ActionResult) send() {
    data, err := result.Json()
    if err != nil {
        return
    }
    invokeResult(result.callback, string(data))
    if result.Method != messageMethod {
        releaseObject(result.callback)
    }
}
```

**event listener 不 release —— 事件频道是常驻的，其它一次性调用每次都释放**。这是经典的"长生命周期"vs"短生命周期"分流。

### 2.5 result.callback 的清理

`core/lib.go:227-234`：

```go
//export setEventListener
func setEventListener(listener unsafe.Pointer) {
    if eventListener != nil || listener == nil {
        releaseObject(eventListener)
    }
    eventListener = listener
}
```

**重新设置或清空 listener 之前必须释放旧的**，否则多次 `setEventListener` 引起旧 callback 永远无法释放——这就是一个 use-after-leak。

## 三、第 2 类：Dart↔Rust（flutter_rust_bridge）—— 一份自动管理

### 3.1 FRB 自动包了"所有权甜层"

```yaml
# plugins/rust_api/flutter_rust_bridge.yaml:1-3
rust_input: crate::api
rust_root: rust/
dart_output: lib/src/rust
```

配合 `pubspec.yaml:5-22`：

```yaml
dependencies:
  flutter_rust_bridge: 2.12.0

flutter:
  plugin:
    platforms:
      ios:     { ffiPlugin: true }
      linux:   { ffiPlugin: true }
      macos:   { ffiPlugin: true }
      windows: { ffiPlugin: true }
```

FRB codegen 自动产出：

- `lib/src/rust/frb_generated.dart` — `RustLib` singleton
- `lib/src/rust/frb_generated.io.dart` — `DynamicLibrary.open` + `lookup()`
- `lib/src/rust/frb_generated.web.dart` — Web stub

### 3.2 Rust 端"Box::into_raw / Box::from_raw"约定

在 FFI 边界，Rust 通常这样传递所有权：

```rust
// 离开 Rust：交给 Dart
let obj = MyClass { ... };
let ptr = Box::into_raw(Box::new(obj)) as *mut c_void;

// 从 Dart 收回所有权
let obj = unsafe { Box::from_raw(ptr as *mut MyClass) };
drop(obj);  // 走正常 Drop
```

Dart 端用 `RustLib.dispose()` 把所有 `RustFrb` 包装的外部对象从 Dart 侧释放 → 自动触发对 Rust 的 `Box::from_raw` + drop。

这是 flutter_rust_bridge 通过代码生成在每个函数里**自动插入**的。

### 3.3 没有 Android — 不是 bug，是策略

`plugins/rust_api/pubspec.yaml` `plugin.platforms` 列表里**没有 Android**。

因为 Android 走的是更紧密的 CGO（第 1 类），不需要 Rust 在 Android 上做 IPC 抽象。这表明：

> 平台的"应该在哪一层"由 **成本** 决定：CGO 比 Rust+IPC 更紧，所以 Android 选 CGO；其他平台没 Java SDK 重压，所以选 Rust+IPC。

## 四、第 3 类：跨进程 IPC（Unix socket / Named pipe / TCP）—— 经典所有权边界

### 4.1 Dart 端的 transport

`lib/core/transport.dart:9-15`：

```dart
const _typeReady = 0x00;
const _typeConnected = 0x01;
const _typeDisconnected = 0x02;
const _typeData = 0x03;
const _typeError = 0x04;
```

`lib/core/transport.dart:17-30`：

```dart
class IPCCoreTransport {
  final String address;
  final StreamController<Uint8List> _dataController =
      StreamController<Uint8List>();
  StreamSubscription<Uint8List>? _subscription;
  Completer<void> _completer = Completer<void>();
  Completer<void> _readyCompleter = Completer<void>();

  void Function()? onDisconnect;
  IPCCoreTransport({required this.address});
  ...
}
```

**4 类二进制帧类型 + 5 个 completion 流**，Dart 端的所有权是：

- `_dataController`（Stream）由 Dart 持有，由 `close()` 释放
- `_subscription` 引用 Rust helper 的 SendPort / Stream，必须显式 cancel 否则 Riverpod 也不会帮你回收

`lib/core/transport.dart:98-105`：

```dart
Future<void> close() async {
  await _subscription?.cancel();
  _subscription = null;
  await stopIpcServer();
  _readyCompleter = Completer<void>();
  _completer = Completer<void>();
  await _dataController.close();
}
```

典型的 4 步 close：subscription → IPC server → 两个 completer → 控制器。

### 4.2 Go 端的 socket/pipe 同样对称

`core/server.go:12-15`：

```go
var (
    conn   io.ReadWriteCloser
    connMu sync.Mutex
)
```

`connMu` 锁的不只是线程同步，还**防止 `send` race**：

`core/server.go:55-65`：

```go
func send(data []byte) {
    if conn == nil {
        logError("send conn nil")
        return
    }
    connMu.Lock()
    defer connMu.Unlock()
    if err := writeFrame(conn, data); err != nil {
        logError("server write error: %v", err)
    }
}
```

—— 因为 `length-prefixed frame` 必须一次 `write(4+len)` **原子**写两次，长度前缀和内容被切开会引发 `readFrame` 解码错乱。**互斥锁本质上是"避免 IO 半写状态"——内存所有权是 OS 文件描述符、连接那边由两端内核系统协调，但**写入完整性由两端应用各自保证**。

### 4.3 connection lifecycyle

`core/server.go:67-102`：

```go
func startServer(arg string) {
    var err error
    conn, err = dial(arg)
    if err != nil {
        panic(err.Error())
    }
    defer func(conn io.Closer) {
        _ = conn.Close()
    }(conn)

    for {
        data, err := readFrame(conn)
        if err != nil {
            if err != io.EOF {
                logError("server read error: %v", err)
            }
            return
        }
        ...
    }
}
```

**"拿到 conn，立即 defer Close"是 Go 端 socket 的标准所有权表达**——一旦解 dial 成功，之后任何 return path 都会让 conn 被 close。

### 4.4 Dart→Go 的 payload：length-prefixed frame

`core/server.go:34-53`：

```go
func writeFrame(w io.Writer, data []byte) error {
    frame := make([]byte, 4+len(data))
    binary.LittleEndian.PutUint32(frame, uint32(len(data)))
    copy(frame[4:], data)
    _, err := w.Write(frame)
    return err
}

func readFrame(r io.Reader) ([]byte, error) {
    lenBuf := make([]byte, 4)
    if _, err := io.ReadFull(r, lenBuf); err != nil {
        return nil, err
    }
    length := binary.LittleEndian.Uint32(lenBuf)
    data := make([]byte, length)
    if _, err := io.ReadFull(r, data); err != nil {
        return nil, err
    }
    return data, nil
}
```

`io.ReadFull` 的必要性：partial read 不可能失败，必须 `io.ReadFull` 才能拿到完整长度。**这一段是 IPC 防泄露 / 防错位的经典**：

- 如果只读 1 字节就读 length → buffer 漂移
- 如果写一半 `Write` 返回 → 对端 `readFrame` 永远卡住

### 4.5 进程崩溃的对端处理

`lib/core/service.dart`（约第 41–60 行）：

```dart
void Function()? onDisconnect;

_transport.onDisconnect = () {
    _handleInvokeCrashEvent();
    if (!_shutdownCompleter.isCompleted) {
        _shutdownCompleter.complete(true);
    }
};
```

Go 进程崩 → pipe / unix socket 返回 EOF → Dart 端 read Frame 返回 `null` → `onDisconnect` 触发 → app 进入 crash UI。

**这是 IPC 边界上"泄露"的副作用：进程本身不存在了，FD 全部 OS 自动回收；但 Dart 侧的 subscription 仍要主动 cancel，否则会跑空 stream。**

## 五、第 4 类：Windows Rust helper（system service）—— 全 token 化

### 5.1 token 设计

`services/helper/Cargo.toml:13-17`：

```toml
[dependencies]
windows-service = { version = "0.7.0", optional = true }
tokio = { version = "1", features = ["full"] }

```

`warp + tokio` 让 helper 自己起一个 http server，Dart 用 dart:io 的 `HttpClient` 短连接调用。**token 用 SHA-256，Dart 端持有同一个 hash，握手时不传值只校验**——一种 zero-knowledge pattern。

### 5.2 build-time 嵌入

`AGENTS.md:192-197`：

```
**Windows helper auth (release):** Core SHA256 is embedded in both the Flutter
app (`--dart-define`) and the Rust helper (`TOKEN` env var during cargo build).
The Dart app pings the helper and verifies the token matches.

**Windows helper auth (debug):** The Rust helper skips token verification
when built in debug mode (`cfg!(debug_assertions)`), so `flutter run` works
without any SHA256 dance.
```

**编译期一致 hash** → 双方都对"我该有的 SHA 是 X"有同等知识，传输时只验证对方**也拥有同样的知识**——不会泄露 hash。这是身份而不是认证。

## 六、跨语言防泄露的 5 条核心原则（每条都有 FlClash 证据）

### 6.1 原则 1：**立刻复制，原指针立刻释放**（避免 share memory）

`core/bride.go:32-36`：

```go
func takeCString(s *C.char) string {
    defer C.free_string(s)
    return C.GoString(s)
}
```

任何一种跨语言传字符串的代码，**最终动作肯定是 `copy + immediate free`**。

### 6.2 原则 2：**桥函数永远不持有跨语言句柄超过必要周期**

`core/lib.go:60-70` + `core/lib.go:162-171`：

```go
func (th *TunHandler) clear() {
    ...
    if th.callback != nil {
        releaseObject(th.callback)
    }
    th.callback = nil
}

func (result ActionResult) send() {
    ...
    invokeResult(result.callback, string(data))
    if result.Method != messageMethod {
        releaseObject(result.callback)
    }
}
```

TUN handler 的 callback → 生命周期明确 → clear 时归还
Result callback → 调用完归还（message 方法除外）

**桥函数永远不直接 delete/TearDown 跨语言对象，必须通过 create side 的 release 函数**。

### 6.3 原则 3：**长生命周期 vs 短生命周期区分**

| 长周期 | 短周期 |
|---|---|
| event listener (`setEventListener`) | 一次性 invoke (`invokeAction`) |
| TUN handler (`TunHandler`) | each `ActionResult` callback |
| event listener 必须重新设置前 release 旧的 (`if eventListener != nil { releaseObject(...) }`) | 短周期在 send 完就 release |
| **绝不能无差别 release** | **绝不能泄漏** |

### 6.4 原则 4：**避免 IPC 半写（partial read/write）**

`core/server.go:34-65` 里 `writeFrame` + `io.ReadFull` + `connMu` 三件套。这一段本质上是把"上层内存所有权"和"OS 文件描述符所有权"绑在一起：

- 长度前缀 + 数据原子写一次
- `io.ReadFull` 强制读完 `length` 字节
- `connMu` 保证多 goroutine 不会交错写半截

### 6.5 原则 5：**自动代码生成胜过手写 FFI**

flutter_rust_bridge 在每个 `#[frb]` 函数自动插入：所有权移交代码（`Box::into_raw`、`Box::from_raw`） + 类型对照 + 错误转译 + 生命周期注释。**手工 dart:ffi 代码总是漏这一步**——上面 `core/lib.go` 的 `releaseObject` / `takeCString` 正是手动管理的手作痕迹。

## 七、跨语言抢占时"潜在泄露点"清单（按 FlClash 实际风险排序）

| 风险点 | FlClash 对策 |
|---|---|
| Dart 端 callback 被 Dart GC 回收 | Go 端持有 `unsafe.Pointer`，TUN 销毁时 `releaseObject` 显式 release |
| Dart→C string 在 Go 端未释放 | `takeCString` 内部 `defer C.free_string`（`core/bride.go:34`） |
| Go→Dart string 在 C 端 free 后被 Dart 异步访问 | Dart 端 `C.result(callback, s)` 是同步 inline，free 在 Dart 拷完才发生 |
| 多次 `setEventListener` 覆盖旧 listener 引发泄露 | `core/lib.go:230` 重新设置前 `releaseObject(eventListener)` |
| IPC 半帧 | `core/server.go:34-53` length-prefixed + `io.ReadFull` + `connMu` |
| Go 子进程崩溃 → Dart 端永远等 | `lib/core/transport.dart:56-60` 的 `onDisconnect` 兜底 |
| Rust helper token 错误 → 持错误身份通信 | `cfg!(debug_assertions)` + SHA-256 zero-knowledge handshake |
| TUN handler 的 callback 在 reconnect 时还是旧的 | 每次 `handleStartTun` 都先 `handleStopTun`（`core/lib.go:141-152`） |
| `cargo + cgo + Go` 复合 build 把 jniLibs 写在多个 Gradle 模块里，复制时 AGP 选错的旧 lib | `AGENTS.md:198-200` 提醒"每次重新 clone + 重 build 修复" |
| Flutter Riverpod providers 跨进程重连时持有旧 Completer | `_callbackCompleterMap` 在 `_initServer` 重新 init 时做互斥初始化 |

## 八、终极抽象：跨语言边界 = IPC 边界 = 序列化字符串边界

把上面 4 类桥观察成 1 类框架：

| 桥形态 | 报文 | 跨语言句柄 / 指针 | 释放时点 |
|---|---|---|---|
| CGO FFI (Go↔Dart/Android) | JSON 字符串 + native pointer 表 | `unsafe.Pointer` 被 Go 端结构体长期持有 | 显式 `releaseObject` + TUN 周期结束 |
| flutter_rust_bridge | 自动 generate, 多种基本类型 + string | `Box::into_raw` 出 Rust → Dart 侧 `NativePtr` 持有 | `RustLib.dispose()` 触发 `Box::from_raw` |
| IPC 子进程 | JSON over length-prefixed frame | 无（OS 文件描述符） | `close(conn)` / `subscription.cancel()` |
| Rust helper + HTTP | HTTP + JSON + SHA-256 | 无（http client / warp server） | 每个 request-response 周期回收 |

**共同点**：

- 任何时候跨语言边界，**长生命周期句柄必须被显式 tracked + manual release**
- 任何时候字符串跨语言，**总是值传递（copy）而非引用**
- 任何时候原生资源跨进程，**必须有进程级 close protocol**

FlClash 的 4 类桥接都坚持了这一套，这就是跨语言栈能"5 平台稳定"的真正原因。
