# IP 多播协议

## 概述

IP 多播（IP Multicast）是 IP 层的组播通信技术，实现 **"一对多"** 的数据分发。

```
RFC 1112 定义了 IP 多播的标准
所有主流操作系统和路由器都支持
```

---

## 地址分类

| 范围 | 类型 | 用途 |
|------|------|------|
| `1.0.0.0` - `126.255.255.255` | A 类单播 | 公共互联网 |
| `128.0.0.0` - `191.255.255.255` | B 类单播 | 企业/组织 |
| `192.0.0.0` - `223.255.255.255` | C 类单播 | 局域网 |
| **224.0.0.0 - 239.255.255.255** | **D 类多播** | **一对多通信** |
| `240.0.0.0` - `255.255.255.255` | E 类保留 | 实验用 |

---

## 多播地址细分

### 本地链路多播 (224.0.0.0/24)
**路由器不转发**，只在同一物理网络传播。

| 地址 | 含义 |
|------|------|
| `224.0.0.1` | 所有主机（同一网段所有设备） |
| `224.0.0.2` | 所有路由器 |
| `224.0.0.251` | DNS 多播 |
| `224.0.0.167` | LocalNet/LocalSend 协议 |

### 用户可用多播 (224.0.1.0 - 231.255.255.255)
可跨路由器传播，需要 IGMP/PIM 协议支持。

### 特定用途多播
| 地址 | 用途 |
|------|------|
| `224.0.0.9` | RIP v2 |
| `224.0.0.13` | PIM 路由协议 |
| `232.0.0.0/8` | SSM (单源多播) |

## 通信模式对比

```
单播 (1对1)：
  A ──────→ B
  "给你"

广播 (1对所有)：
  A ──┬──→ B
      ├──→ C
      └──→ D
  "给所有人"

多播 (1对一组)：
  A ──┬──→ B
      ├──→ C    (D 不要)
      └──→ X
  "给 B C X"
```

| 特性 | 单播 | 广播 | 多播 |
|------|------|------|------|
| 目标 | 单个设备 | 所有设备 | 加入组的设备 |
| 需要知道对方 IP | 是 | 否 | 否（只知道组地址） |
| D 收到 A 的广播？ | - | 会 | 不会 |
| 跨路由器 | 直接支持 | 通常不支持 | 需要多播路由 |

## 操作系统支持

### 核心 API

```dart
// Dart (dart:io)
socket.joinMulticast(InternetAddress('224.0.0.167'));
socket.leaveMulticast(InternetAddress('224.0.0.167'));

// C
int setsockopt(int sockfd, IPPROTO_IP, IP_ADD_MEMBERSHIP,
               &mreq, sizeof(mreq));

// Linux
ip maddr add 224.0.0.167 dev eth0

// Windows
setsockopt(sock, IPPROTO_IP, IP_ADD_MEMBERSHIP, ...);
```

### 操作系统内部流程

```
应用层:  joinMulticast(224.0.0.167)
           │
           ▼
操作系统:  在 NIC 驱动层注册"我要收 224.0.0.167 的包"
           │
           ▼
网卡:      硬件过滤 - 只接收目标为 224.0.0.167 的帧
           │
           ▼
收到数据包 → 操作系统组装 → 应用 recv()
```

## 协议栈位置

```
┌─────────────────────────────────┐
│         应用层                   │  ← HTTP, FTP, 你的业务协议
├─────────────────────────────────┤
│         传输层 (TCP/UDP)         │  ← 多播通常用 UDP
├─────────────────────────────────┤
│         IP 层                   │  ← IP_MULTICAST (224.x.x.x)
├─────────────────────────────────┤
│       数据链路层                 │  ← Ethernet 多播 (01:00:5e:xx:xx:xx)
├─────────────────────────────────┤
│         物理层                   │
└─────────────────────────────────┘

多播地址到 MAC 地址的映射：
224.0.0.167 → 01:00:5e:00:00:67 (MAC)
              │         │ └─ 低 23 位
              └───────────── 固定前缀 (01:00:5e)
```

## 多播地址的访问逻辑

### 核心概念：频道号，不是真实 IP

```
224.0.0.167 = "频道 167"
没有人拥有这个 IP
它不是一个设备的地址
```

### 访问流程图

```
┌─────────────────────────────────────────────────────────────┐
│                        网络                                  │
│                                                             │
│  设备 A (192.168.1.10)                                      │
│    ├─ join 224.0.0.167  ←── 订阅频道                        │
│    └─ socket.send(data, 224.0.0.167, 53317)                 │
│                           │                                  │
│                           ▼                                  │
│  交换机 ──────────────────────────────────────────────────  │
│    │                                                        │
│    │  "谁订阅了 224.0.0.167？"                              │
│    │                                                        │
│    ├─→ 设备 A ✓ (已订阅)                                    │
│    ├─→ 设备 B ✓ (已订阅) ←── join 224.0.0.167               │
│    └─→ 设备 C ✗ (未订阅)                                    │
│                                                             │
│  设备 B (192.168.1.20)                                      │
│    └─ recv() ←── 收到 A 发的数据                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 代码层面的逻辑

```dart
// 1. 设备 B：订阅频道
socket.joinMulticast(InternetAddress('224.0.0.167'));
// 操作系统内核记录："设备 B 要收 224.0.0.167 的包"
// 交换机学习："224.0.0.167 → 端口 B"

// 2. 设备 A：发送消息
socket.send(data, InternetAddress('224.0.0.167'), 53317);
// 交换机查看 MAC 表，发现 224.0.0.167 → 端口 B
// 转发数据到设备 B

// 3. 设备 B：收到消息
socket.listen((event) {
  if (event == RawSocketEvent.read) {
    recv(); // 收到 A 发来的数据
  }
});
```

### IP 到 MAC 的映射

```
224.0.0.167 是 IP 地址
但 Ethernet 帧需要 MAC 地址
所以需要映射：

IP:   224.0.0.167
MAC:  01:00:5e:00:00:67
       │         │ └─ 低 23 位
       └──────────── 固定前缀 (01:00:5e)

映射算法：
224 = 11100000
0   = 00000000
0   = 00000000
167 = 10100111

取低 23 位: 00 0000 1010 0111 = 0x00a7
MAC 前缀:   01:00:5e:00:00:a7
```

### 网卡层面

```
普通 Unicast：
  "只接收目标 MAC == 我的 MAC 的帧"

Multicast：
  "接收目标 MAC == 01:00:5e:xx:xx:xx 的帧"
  且 IP 层过滤只看 224.x.x.x

所以多播效率高：
  - 网卡硬件过滤，不是每帧都中断 CPU
  - 交换机按需转发，不浪费带宽
```

### 完整数据流

```
设备 A 发送 "Hello" 到 224.0.0.167:53317
│
├─ 应用层：socket.send(data, "224.0.0.167", 53317)
│
├─ 传输层：UDP 添加源/目的端口
│
├─ IP 层：
│   目的 IP = 224.0.0.167 (多播地址)
│   协议号 = 17 (UDP)
│
├─ Ethernet 层：
│   目的 MAC = 01:00:5e:00:00:67 (多播 MAC)
│   类型 = 0x0800 (IPv4)
│
├─ 交换机：
│   查 MAC 表："224.0.0.167 → 端口 B, 端口 C"
│   复制帧转发到端口 B 和 C
│
├─ 设备 B 网卡：
│   硬件过滤：检查目的 MAC 是否为 01:00:5e:xx:xx:xx
│   匹配！将帧交给操作系统
│
└─ 设备 B 应用层：
    socket.recv() → "Hello"
```

### 一句话总结

> **224.0.0.167 是一个"逻辑频道号"。加入这个组（joinMulticast）的设备会收到发往这个地址的所有数据。没加入的设备完全感知不到这个地址的存在。**

## 网络设备行为

### 交换机
```
A 发送 → 224.0.0.167
  │
  ▼
交换机学习："谁加入了这个多播组"
  │
  ├→ 端口 1 (B 加入了) ✓ 转发
  ├→ 端口 3 (C 加入了) ✓ 转发
  └→ 端口 5 (D 没加入) ✗ 不转发
```

### 路由器
```
多播包跨网段需要：
1. IGMP (Internet Group Management Protocol) - 主机注册
2. PIM (Protocol Independent Multicast) - 路由协议

224.0.0.0/24 范围路由器不转发
224.0.1.0+   范围可跨路由器
```

## 与订阅/发布模式的区别

| | IP 多播 | 订阅/发布 |
|---|---|---|
| **层次** | 网络层（操作系统） | 应用层 |
| **标准化** | 全球统一 (RFC 1112) | 应用自定义 |
| **实现** | 操作系统/路由器 | 代码实现 |
| **"订阅"** | `joinMulticast()` 一次 | 每次消息都要路由 |
| **内容** | 任意二进制数据 | 业务消息 |

**精神相似**：
```
订阅/发布：
  subscribe("topic")
  publish("topic", data)

IP 多播：
  join_multicast_group(addr)      ← 订阅
  send_to_multicast_group(addr)   ← 发布
```

**区别**：
- IP 多播的"订阅"是操作系统级别的，一次加入，持续接收
- 应用层订阅/发布每次消息都要 broker 路由

## 示例：LocalNet 发现协议

```dart
class DiscoveryService {
  static const String multicastAddress = '224.0.0.167';
  static const int multicastPort = 53317;

  RawDatagramSocket? _udpSocket;

  Future<void> startListening() async {
    // 1. 创建 UDP socket
    _udpSocket = await RawDatagramSocket.bind(
      InternetAddress.anyIPv4,
      multicastPort,
      reuseAddress: true,
      reusePort: true,
    );

    // 2. 加入多播组（订阅）
    _udpSocket!.joinMulticast(InternetAddress(multicastAddress));

    // 3. 监听收到的消息
    _udpSocket!.listen((event) {
      if (event == RawSocketEvent.read) {
        final datagram = _udpSocket!.receive();
        // 处理收到的多播消息
      }
    });
  }

  // 4. 发送广播（发布）
  void sendBroadcast() {
    final data = utf8.encode('$deviceId,$port');
    _udpSocket!.send(
      data,
      InternetAddress(multicastAddress),
      multicastPort,
    );
  }
}
```

## 限制与注意事项

| 限制 | 说明 |
|------|------|
| **本地链路** (`224.0.0.0/24`) | 路由器不转发，只能同一网段 |
| **NAT** | 多播穿透 NAT 困难，需要额外配置 |
| **丢包** | UDP 本身不可靠，多播也一样 |
| **顺序** | UDP 不保证顺序 |
| **防火墙** | 需要开放 UDP 多播端口 |

## 总结

```
IP 多播 = 网络层的基础设施
         │
         ├─ 224.x.x.x 是全球标准的"组播地址段"
         ├─ joinMulticast() = 订阅某个组
         ├─ send() 到组地址 = 向所有订阅者广播
         └─ 路由器/交换机负责按组分发

多播是"一对多"的最优解：
  比单播高效（不用 N 次发送）
  比广播精准（只发给订阅者）
  比应用层 broker 轻量（操作系统直接支持）
```
