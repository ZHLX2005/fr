## 概述

IP 多播（IP Multicast）是 IP 层的组播通信技术，实现 **"一对多"** 的数据分发。

```
RFC 1112 定义了 IP 多播的标准
所有主流操作系统和路由器都支持
```

## 地址分类

| 范围 | 类型 | 用途 |
|------|------|------|
| `1.0.0.0` - `126.255.255.255` | A 类单播 | 公共互联网 |
| `128.0.0.0` - `191.255.255.255` | B 类单播 | 企业/组织 |
| `192.0.0.0` - `223.255.255.255` | C 类单播 | 局域网 |
| **224.0.0.0 - 239.255.255.255** | **D 类多播** | **一对多通信** |
| `240.0.0.0` - `255.255.255.255` | E 类保留 | 实验用 |

## 多播地址细分

### 本地链路多播 (224.0.0.0/24)
**路由器不转发**，只在同一物理网络传播。

| 地址 | 含义 |
|------|------|
| `224.0.0.1` | 所有主机（同一网段所有设备） |
| `224.0.0.2` | 所有路由器 |
| `224.0.0.251` | DNS 多播 |
| `224.0.0.167` | LocalNet/LocalSend 协议 |

### 用户可用多播 (224.0.1.0 - 231.255.255.255)
可跨路由器传播，需要 IGMP/PIM 协议支持。

### 特定用途多播
| 地址 | 用途 |
|------|------|
| `224.0.0.9` | RIP v2 |
| `224.0.0.13` | PIM 路由协议 |
| `232.0.0.0/8` | SSM (单源多播) |

## 多播 IP 存在于哪里？

### 核心结论：存在于 IP 层，不是"真实"的 IP

```
单播 IP（如 192.168.1.100）：
  存在于 IP 层
  是一台真实设备在网络中的标识
  网卡有真实的 IP 地址

多播 IP（如 224.0.0.167）：
  存在于 IP 层
  不是一个设备的标识，是一个"组"的标识
  没有任何设备"拥有"这个 IP
```

### 类比

```
单播 IP = 手机号码
  分配给某个人的设备
  别人打这个号码就能找到这个人

多播 IP = 频道号
  不是分配给谁的
  谁调到 107.9 MHz 就能听到广播
  没有"主人"
```

## TCP 能使用多播地址吗？

### 答案：**不能**

```
TCP 是面向连接的协议
  → 需要建立"点对点"的连接
  → 客户端 connect() → 服务器 accept()

多播是一对多的通信
  → 一个发送者，多个接收者
  → 无法建立"一对一"的 TCP 连接
```

### 协议栈层次

```
┌─────────────────────────────────────┐
│  应用层 (HTTP, FTP, 你的 App)        │
├─────────────────────────────────────┤
│  传输层                             │
│   ├─ TCP  ─── 面向连接、一对一      │ ← 不能用于多播
│   └─ UDP  ─── 无连接、一对多        │ ← 支持多播
├─────────────────────────────────────┤
│  IP 层                             │
│   ├─ 单播 (192.168.1.x) ─ 点对点    │
│   └─ 多播 (224.x.x.x) ─ 一对多      │ ← IP 层的功能
└─────────────────────────────────────┘

结论：多播是 IP 层的特性，但通常只配合 UDP 使用
```

### 为什么 TCP 不能多播？

```
场景：TCP 服务器监听 224.0.0.167:8080

问题 1：TCP 需要 accept() 建立连接
        → 多个接收者来临，哪个建立连接？

问题 2：TCP 是可靠传输，需要确认
        → 发送方如何确认"一组"接收者都收到了？

问题 3：TCP 连接是"一对一"的
        → 多播是一对多，协议设计不兼容
```

### 多播只支持 UDP

```
┌─────────────────────────────────────────────┐
│              多播支持的协议                   │
├─────────────────────────────────────────────┤
│                                             │
│  UDP + 多播 = ✓                            │
│   ├─ RTP (实时音视频)                       │
│   ├─ mDNS (局域网发现)                      │
│   ├─ SSDP (UPnP 发现)                      │
│   └─ LocalNet 发现协议                      │
│                                             │
│  TCP + 多播 = ✗                            │
│   └─ TCP 本身就不支持多播                   │
│                                             │
└─────────────────────────────────────────────┘
```

### 一句话

> **多播 IP 存在于 IP 层（网络层），是一个逻辑标识，不是真实设备的 IP。TCP 是面向连接的协议，无法使用多播。只有 UDP 才能配合多播工作，因为 UDP 是无连接的，支持一对多通信。**

## 通信模式对比

```
单播 (1对1)：
  A ──────→ B
  "给你"

广播 (1对所有)：
  A ──┬──→ B
      ├──→ C
      └──→ D
  "给所有人"

多播 (1对一组)：
  A ──┬──→ B
      ├──→ C    (D 不要)
      └──→ X
  "给 B C X"
```

| 特性 | 单播 | 广播 | 多播 |
|------|------|------|------|
| 目标 | 单个设备 | 所有设备 | 加入组的设备 |
| 需要知道对方 IP | 是 | 否 | 否（只知道组地址） |
| D 收到 A 的广播？ | - | 会 | 不会 |
| 跨路由器 | 直接支持 | 通常不支持 | 需要多播路由 |

## 操作系统支持

### 核心 API

```dart
// Dart (dart:io)
socket.joinMulticast(InternetAddress('224.0.0.167'));
socket.leaveMulticast(InternetAddress('224.0.0.167'));

// C
int setsockopt(int sockfd, IPPROTO_IP, IP_ADD_MEMBERSHIP,
               &mreq, sizeof(mreq));

// Linux
ip maddr add 224.0.0.167 dev eth0

// Windows
setsockopt(sock, IPPROTO_IP, IP_ADD_MEMBERSHIP, ...);
```

### 操作系统内部流程

```
应用层:  joinMulticast(224.0.0.167)
           │
           ▼
操作系统:  在 NIC 驱动层注册"我要收 224.0.0.167 的包"
           │
           ▼
网卡:      硬件过滤 - 只接收目标为 224.0.0.167 的帧
           │
           ▼
收到数据包 → 操作系统组装 → 应用 recv()
```

## 端口存在的意义

### 问题：一台设备上可能有多个应用需要网络通信

```
192.168.1.100 (我的电脑)
  ├─ 浏览器要访问网页
  ├─ 邮件客户端要收邮件
  ├─ LocalNet 要发现设备
  └─ 视频软件要通话

收到数据了，怎么知道给哪个应用？
```

### 端口 = 应用程序的"邮箱编号"

```
IP 地址 = 大楼地址（设备）
端口号  = 房间号（应用）

192.168.1.100:53317
│               └──── LocalNet 的"房间号"
└──────────────────── 电脑的"地址"
```

### 类比：港口/码头

```
IP地址：城市
端口号： 港口编号

  上海港 ──────── 货物交给谁？
    │              │
    ├─ 1号泊位 ──→ 集装箱（Web服务）
    ├─ 2号泊位 ──→ 散货（邮件服务）
    └─ 3号泊位 ──→ 油轮（LocalNet服务）
```

### 常见端口约定

| 端口 | 服务 |
|------|------|
| 80 | HTTP 网页 |
| 443 | HTTPS 加密网页 |
| 53 | DNS |
| 25 | SMTP 邮件发送 |
| 110 | POP3 邮件接收 |
| 53317 | LocalNet 发现协议 |

### 端口是"端到端"的概念

```
单播通信：
  A:192.168.1.10:5000  ──────→  B:192.168.1.20:8080
   [IP:端口] 发送方            [IP:端口] 接收方
                              ↑
                              这就是"端"

多播通信：
  A 发到 224.0.0.167:53317
  │
  ├─ B 收到 (B 的 socket 绑定了 53317)
  └─ C 收到 (C 的 socket 绑定了 53317)
       ↑
       53317 是"目标端"
```

### 为什么端口是必要的？

```
场景：只用 IP，不分层

"192.168.1.100 给你发数据了"
  → 浏览器？邮件？LocalNet？
  → 不知道！冲突了！

场景：IP + 端口分层

"192.168.1.100:53317 给你发数据了"
  → LocalNet 应用知道"这是给我的"
  → 端口区分了不同的服务/应用
```

### 一句话

> **端口是操作系统用来区分同一设备上不同应用/服务的编号。IP 地址找到设备，端口找到该设备上的具体应用。**

## 订阅 IP 还是 IP+端口？

### 核心结论：订阅的是 IP 地址，不是 IP+端口

| 层 | 订阅/加入 | 目的 |
|----|----------|------|
| **IP 层** | `joinMulticast(224.0.0.167)` | 订阅"频道" |
| **传输层** | `bind(53317)` | 绑定"端口" |

**IP 多播是 IP 层的机制**，它只管把数据从 A 发给一组人。端口是 UDP 在 IP 之上的概念。

### `224.0.0.167:53317` 的含义

```
224.0.0.167:53317
│              │
│              └── 端口（UDP/传输层）← 应用绑定
│
└── 组地址（IP/网络层）← 多播订阅
```

**这是两个正交的概念**：

```
IP 层：   "发给 224.0.0.167 这个组"     ← 谁会收到
UDP 层：  "用 53317 这个端口"           ← 收到后交给哪个 socket
```

### 收发流程对比

```
单播：     A:192.168.1.10:5000  ──────→  B:192.168.1.20:8080
           [IP:端口] 发送方         [IP:端口] 接收方
           IP 对 IP，端口对端口

多播：     A:192.168.1.10:5000  ──┬──→  B:192.168.1.20:xxxx
                                 └──→  C:192.168.1.30:xxxx
           [IP:端口] 发送方          IP 对 IP（组播地址）
                                    端口对端口（各自绑定）
```

### 示例：两个应用在同一设备

```dart
// 两个应用都加入多播组 224.0.0.167，但绑定不同端口

App A:                    App B:
joinMulticast(224.0.0.167)   joinMulticast(224.0.0.167)
bind(53317)                 bind(53318)

// A 发到 224.0.0.167:53317
socket.send(data, 224.0.0.167, 53317)
//                    │         │
//                    │         └── 只给端口 53317 的 socket
//                    └── 发到 IP 组（但 UDP 层会过滤端口）
```

### 实际数据流

```
A 发: "Hello" → 224.0.0.167:53317

IP 层：
  发往 224.0.0.167
  交换机复制 → 所有订阅者都收到

UDP 层（每个接收者）：
  检查"目标端口是我绑定的端口吗？"
  53317 → App A 收到
  53318 → App B 收到（端口不匹配，丢弃）
```

### 一句话总结

> **多播订阅的是 IP 地址（频道），端口是每个应用独立绑定的。发到 `224.0.0.167:53317` 意思是"发到多播组，目标是端口 53317"。只有绑定 53317 端口的应用才会收到。**

## 多播地址的访问逻辑

### 核心概念：频道号，不是真实 IP

```
224.0.0.167 = "频道 167"
没有人拥有这个 IP
它不是一个设备的地址
```

### 访问流程图

```
┌─────────────────────────────────────────────────────────────┐
│                        网络                                  │
│                                                             │
│  设备 A (192.168.1.10)                                      │
│    ├─ join 224.0.0.167  ←── 订阅频道                        │
│    └─ socket.send(data, 224.0.0.167, 53317)                 │
│                           │                                  │
│                           ▼                                  │
│  交换机 ──────────────────────────────────────────────────  │
│    │                                                        │
│    │  "谁订阅了 224.0.0.167？"                              │
│    │                                                        │
│    ├─→ 设备 A ✓ (已订阅)                                    │
│    ├─→ 设备 B ✓ (已订阅) ←── join 224.0.0.167               │
│    └─→ 设备 C ✗ (未订阅)                                    │
│                                                             │
│  设备 B (192.168.1.20)                                      │
│    └─ recv() ←── 收到 A 发的数据                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 代码层面的逻辑

```dart
// 1. 设备 B：订阅频道
socket.joinMulticast(InternetAddress('224.0.0.167'));
// 操作系统内核记录："设备 B 要收 224.0.0.167 的包"
// 交换机学习："224.0.0.167 → 端口 B"

// 2. 设备 A：发送消息
socket.send(data, InternetAddress('224.0.0.167'), 53317);
// 交换机查看 MAC 表，发现 224.0.0.167 → 端口 B
// 转发数据到设备 B

// 3. 设备 B：收到消息
socket.listen((event) {
  if (event == RawSocketEvent.read) {
    recv(); // 收到 A 发来的数据
  }
});
```

### IP 到 MAC 的映射

```
224.0.0.167 是 IP 地址
但 Ethernet 帧需要 MAC 地址
所以需要映射：

IP:   224.0.0.167
MAC:  01:00:5e:00:00:67
       │         │ └─ 低 23 位
       └──────────── 固定前缀 (01:00:5e)

映射算法：
224 = 11100000
0   = 00000000
0   = 00000000
167 = 10100111

取低 23 位: 00 0000 1010 0111 = 0x00a7
MAC 前缀:   01:00:5e:00:00:a7
```

### 网卡层面

```
普通 Unicast：
  "只接收目标 MAC == 我的 MAC 的帧"

Multicast：
  "接收目标 MAC == 01:00:5e:xx:xx:xx 的帧"
  且 IP 层过滤只看 224.x.x.x

所以多播效率高：
  - 网卡硬件过滤，不是每帧都中断 CPU
  - 交换机按需转发，不浪费带宽
```

### 完整数据流

```
设备 A 发送 "Hello" 到 224.0.0.167:53317
│
├─ 应用层：socket.send(data, "224.0.0.167", 53317)
│
├─ 传输层：UDP 添加源/目的端口
│
├─ IP 层：
│   目的 IP = 224.0.0.167 (多播地址)
│   协议号 = 17 (UDP)
│
├─ Ethernet 层：
│   目的 MAC = 01:00:5e:00:00:67 (多播 MAC)
│   类型 = 0x0800 (IPv4)
│
├─ 交换机：
│   查 MAC 表："224.0.0.167 → 端口 B, 端口 C"
│   复制帧转发到端口 B 和 C
│
├─ 设备 B 网卡：
│   硬件过滤：检查目的 MAC 是否为 01:00:5e:xx:xx:xx
│   匹配！将帧交给操作系统
│
└─ 设备 B 应用层：
    socket.recv() → "Hello"
```

### 一句话总结

> **224.0.0.167 是一个"逻辑频道号"。加入这个组（joinMulticast）的设备会收到发往这个地址的所有数据。没加入的设备完全感知不到这个地址的存在。**

## 网络设备行为

### 交换机
```
A 发送 → 224.0.0.167
  │
  ▼
交换机学习："谁加入了这个多播组"
  │
  ├→ 端口 1 (B 加入了) ✓ 转发
  ├→ 端口 3 (C 加入了) ✓ 转发
  └→ 端口 5 (D 没加入) ✗ 不转发
```

### 路由器
```
多播包跨网段需要：
1. IGMP (Internet Group Management Protocol) - 主机注册
2. PIM (Protocol Independent Multicast) - 路由协议

224.0.0.0/24 范围路由器不转发
224.0.1.0+   范围可跨路由器
```

## 与订阅/发布模式的区别

| | IP 多播 | 订阅/发布 |
|---|---|---|
| **层次** | 网络层（操作系统） | 应用层 |
| **标准化** | 全球统一 (RFC 1112) | 应用自定义 |
| **实现** | 操作系统/路由器 | 代码实现 |
| **"订阅"** | `joinMulticast()` 一次 | 每次消息都要路由 |
| **内容** | 任意二进制数据 | 业务消息 |

**精神相似**：
```
订阅/发布：
  subscribe("topic")
  publish("topic", data)

IP 多播：
  join_multicast_group(addr)      ← 订阅
  send_to_multicast_group(addr)   ← 发布
```

**区别**：
- IP 多播的"订阅"是操作系统级别的，一次加入，持续接收
- 应用层订阅/发布每次消息都要 broker 路由

## mDNS (Multicast DNS)

### 是什么？

```
mDNS = 多播 DNS
基于 224.0.0.251:5353 进行局域网设备发现
不需要 DNS 服务器
```

### 对比传统 DNS

| | 传统 DNS | mDNS |
|---|---|---|
| **地址** | DNS 服务器 IP | `224.0.0.251:5353` |
| **查询方式** | 问服务器 | 广播问所有人 |
| **用途** | 全局域名 | 局域网域名 |
| **域名后缀** | `.com`、`.org` | `.local` |
| **需要服务器** | 是 | 否 |

### 类比

```
传统 DNS：
  你访问 "google.com"
  → 问 DNS 服务器："google.com 的 IP 是什么？"

mDNS：
  你在局域网内
  → 直接喊："谁有 apple-tv.local？"
  → Apple TV 回答："是我！192.168.1.50"
  → 不需要 DNS 服务器
```

### 工作原理

```
设备 A 想找 "apple-tv.local"
  │
  ▼
A ──mDNS 广播──→ 224.0.0.251:5353
                      │
                      ▼
              交换机复制到所有订阅者
                      │
                      ▼
              设备 B (Apple TV) 收到
                      │
                      ▼
              B 回答："apple-tv.local = 192.168.1.100"
                      │
                      ▼
A ←───mDNS 响应──────
A 知道 B 的 IP 了
```

### 常见 mDNS 服务

| 服务 | 域名后缀 | 用途 |
|------|----------|------|
| Chromecast | `.chromecast` | 投屏设备发现 |
| Apple TV | `.local` | AirPlay 发现 |
| HomePod | `.local` | 音频设备发现 |
| Printer | `.local` | 网络打印机发现 |
| SSH/SFTP | `.local` | 局域网 SSH |

### 一句话

> **mDNS 是基于多播的局域网域名解析协议，设备通过广播问"谁有这个名字？"，拥有该名字的设备回答 IP。不需要 DNS 服务器，适合家庭/办公室局域网。**

## 示例：LocalNet 发现协议

```dart
class DiscoveryService {
  static const String multicastAddress = '224.0.0.167';
  static const int multicastPort = 53317;

  RawDatagramSocket? _udpSocket;

  Future<void> startListening() async {
    // 1. 创建 UDP socket
    _udpSocket = await RawDatagramSocket.bind(
      InternetAddress.anyIPv4,
      multicastPort,
      reuseAddress: true,
      reusePort: true,
    );

    // 2. 加入多播组（订阅）
    _udpSocket!.joinMulticast(InternetAddress(multicastAddress));

    // 3. 监听收到的消息
    _udpSocket!.listen((event) {
      if (event == RawSocketEvent.read) {
        final datagram = _udpSocket!.receive();
        // 处理收到的多播消息
      }
    });
  }

  // 4. 发送广播（发布）
  void sendBroadcast() {
    final data = utf8.encode('$deviceId,$port');
    _udpSocket!.send(
      data,
      InternetAddress(multicastAddress),
      multicastPort,
    );
  }
}
```

## 限制与注意事项

| 限制 | 说明 |
|------|------|
| **本地链路** (`224.0.0.0/24`) | 路由器不转发，只能同一网段 |
| **NAT** | 多播穿透 NAT 困难，需要额外配置 |
| **丢包** | UDP 本身不可靠，多播也一样 |
| **顺序** | UDP 不保证顺序 |
| **防火墙** | 需要开放 UDP 多播端口 |

## UDP 服务器启动：bind vs listen

### 核心区别

| | TCP | UDP |
|---|---|---|
| **绑定端口** | `bind(port)` | `bind(port)` |
| **监听连接** | `listen()` | **不需要** |
| **接受连接** | `accept()` | 不需要 |
| **接收数据** | `read()` | `receive()` |
| **连接状态** | 有连接 (面向连接) | 无连接 |

### 为什么 UDP 不需要 listen？

```
TCP 是"打电话"：
  1. 对方要接听 (listen)
  2. 对方要accept建立连接
  3. 然后才能说话

UDP 是"寄快递"：
  1. 直接发出去
  2. 收件人不需要"接听"
  3. 没有连接建立过程
```

### 启动 UDP 服务器

```dart
// Dart - 只需要 bind，不需要 listen
final socket = await RawDatagramSocket.bind(
  InternetAddress.anyIPv4,
  53317,  // 端口
  reuseAddress: true,
  reusePort: true,  // 允许多个 socket 绑定同一个端口
);

socket.listen((event) {
  if (event == RawSocketEvent.read) {
    final datagram = socket.receive();
    // 处理数据
  }
});
```

### 对比 TCP 服务器

```dart
// TCP - 需要 listen
server = await ServerSocket.bind(port: 53317);
server.listen((socket) {
  // 处理连接
});

// UDP - 只需要 bind
socket = await RawDatagramSocket.bind(port: 53317);
socket.listen((event) {
  // 处理数据报
});
```

### 端口占用与 reusePort

```
正常情况：一个端口只能被一个应用绑定
         ↓
应用 A bind(53317)  成功
应用 B bind(53317)  失败！端口已被占用

reusePort: true：允许多个 socket 绑定同一个端口
           (需要操作系统支持)
           ↓
应用 A bind(53317, reusePort: true)  成功
应用 B bind(53317, reusePort: true)  成功
         ↓
这时候收到数据，两个应用都能收到！
```

### 完整 UDP 服务器示例

```dart
class UdpServer {
  RawDatagramSocket? _socket;

  Future<void> start() async {
    // 绑定端口
    _socket = await RawDatagramSocket.bind(
      InternetAddress.anyIPv4,
      53317,
      reuseAddress: true,
      reusePort: true,  // 允许多个 socket 绑定同一端口
    );

    print('UDP 服务器启动，监听端口: ${_socket!.port}');

    // 加入多播组
    _socket!.joinMulticast(InternetAddress('224.0.0.167'));

    // 监听数据
    _socket!.listen((event) {
      if (event == RawSocketEvent.read) {
        final datagram = _socket!.receive();
        if (datagram != null) {
          _handleDatagram(datagram);
        }
      }
    });
  }

  void _handleDatagram(Datagram datagram) {
    final message = utf8.decode(datagram.data);
    final address = datagram.address.address;
    print('收到: $message (来自 $address)');
  }

  void stop() {
    _socket?.close();
    _socket = null;
  }
}
```

### 一句话

> **UDP 服务器不需要 listen，只需要 `bind(端口)`。bind 是把 socket 和端口绑定，listen 是 TCP 特有的"监听连接"操作，因为 UDP 是无连接的。**

## 多端口监听：广播模式 vs 消费模式

### 核心结论：广播模式

```
多个应用绑定同一端口 (reusePort: true)：
  每个应用都能收到完整的数据副本
  消息不会被"消费掉"
```

### 对比：消费模式 vs 广播模式

```
消息队列（消费模式）：
  ┌─────────────┐
  │  消息队列    │ ← 消息只有一份
  └──────┬──────┘
         │
    ┌────┴────┐
    ▼         ▼
  应用 A    应用 B
  收到后    只有等 A 不想要了
  消息没了   才能收到

UDP reusePort（广播模式）：
  ┌─────────────────────────────────┐
  │         网络                     │ ← 数据包只有一份
  └─────────────┬───────────────────┘
                 │
        ┌────────┴────────┐
        ▼                 ▼
      应用 A            应用 B
      收到副本 A        收到副本 B
      两份数据          两份数据
```

### 典型场景：mDNS

```
设备上有两个应用都监听 5353 端口 (mDNS)
应用 A：Chrome 的 DNS 解析
应用 B：系统的网络发现

收到多播包时：
  Chrome 收到一份 → 处理 DNS 查询
  系统收到一份   → 处理网络发现
  两人都能正常工作
```

### 代码验证

```dart
// 应用 A
socket = RawDatagramSocket.bind(53317, reusePort: true);
socket.listen((e) {
  print('A 收到: ${socket.receive()?.data}');
});

// 应用 B (同一设备)
socket = RawDatagramSocket.bind(53317, reusePort: true);
socket.listen((e) {
  print('B 收到: ${socket.receive()?.data}');
});

// 外部发一个包
// A 收到一份
// B 收到一份
// 两人都收到完整数据！
```

### 消息模式对比

| 模式 | 特点 | 示例 |
|------|------|------|
| **消费模式** | 消息只有一份，谁抢到谁收 | TCP、消息队列 (Kafka) |
| **广播模式** | 每个订阅者都收到独立副本 | UDP + reusePort |

### 一句话

> **UDP 的 reusePort 是**广播模式**，每个绑定同一端口的应用都能收到独立的数据副本，不存在"谁收到其他人就没了"的情况。**

## 总结

```
IP 多播 = 网络层的基础设施
         │
         ├─ 224.x.x.x 是全球标准的"组播地址段"
         ├─ joinMulticast() = 订阅某个组
         ├─ send() 到组地址 = 向所有订阅者广播
         └─ 路由器/交换机负责按组分发

多播是"一对多"的最优解：
  比单播高效（不用 N 次发送）
  比广播精准（只发给订阅者）
  比应用层 broker 轻量（操作系统直接支持）
```
