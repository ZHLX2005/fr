# 围追堵截（Quoridor 变体）局域网联机：底层实现详解

> 本文档从技术角度完整记录 `surround_game` 模块从单机热座扩展为局域网联机的**架构演进、底层难点与工程决策**，涵盖：localnet 框架集成、Session 状态同步、棋盘翻转与触摸镜像、UDP 多播通信、双源状态机一致性问题、以及持久化与 UX 完善。

## 1. 架构总览

```

│   ├── persistence/          ← 本机 alias / deviceId 持久化
│   ├── profile/              ← aliasDialog 组件

│   ├── lan_client_game_page.dart    ← Client 游戏页面
│   ├── lan_*_protocol_bridge.dart   ← 协议 → 状态机纯函数
│   └── lan_*_view_model.dart        ← 状态机（房间/等待/游戏中/结束）
│
└── surround_game.dart     # barrel 文件
```

### 设计原则

| 原则 | 说明 |
|------|------|
| **UI 层拆分** | `lan/` 与 `local/` 互不 import，仅共享 `engine/` `models/` `widgets/` |
| **隔离 framework** | 业务层（Page / ViewModel）不直接 `import` localnet，全走 `LanServiceAdapter` |
| **不可变值对象** | `GameState` 是 immutable `copyWith` 值对象；`ValueNotifier<GameState>` 外层提供 Listenable |
| **TDD 驱动** | 每 task 先写测试 → 红灯 → 实现 → 绿灯 → commit |
| **YAGNI** | 每轮明确不做的范围（SettingsPage、重连、自动 e2e） |

## 2. 游戏引擎与数据模型

### 不可变 GameState

`GameState` 是整个系统的轴心。它是有 10 个字段的不可变值对象：

```
topPlayerId / bottomPlayerId — 双方棋子位置（cellId, 0..80）
currentPlayerIsTop            — 当前回合属哪方
topWallsPlaced / bottomWalls — 各用了几堵墙
adjacency / wallGrid          — 邻接表（81组）/ 墙占用（289格）
history                        — 棋谱栈（MoveRecord 序列）
validMoves                    — 当前玩家合法落子集合
status                        — GameStatus { running, topWin, bottomWin, draw }
```

每次 `movePiece()` 或 `placeWall()` 都返回**新的** `GameState` 实例（`copyWith` 链），绝不变异原实例。这对 LAN 模式的数据流至关重要——线程安全、无共享状态、可自由序列化传输。

### 序列化契约

```dart
// GameState.toJson() —— 序列化所有**用户定义状态**
{
  'topPlayerId': 4,
  'bottomPlayerId': 76,
  'currentPlayerIsTop': true,
  'history': [...MoveRecord...],
  ...
  // 注意：adjacency 和 wallGrid 不序列化！！
}

// GameState.fromJson() —— 反序列化后 adjacency/wallGrid 为空
// 调用方必须：QuoridorEngine.replayHistory(temp.history) 重建
```

这一契约是整个 LAN 同步的基础——对方的游戏状态通过**重放棋谱**重建，而不是直接传输邻接表（邻接表占 81×Set，传输代价高）。

## 3. 状态机设计（sealed class 层次）

### 为什么用 sealed class

Dart 3 的 `sealed class` 提供了**编译期穷尽检查**：

```dart
sealed class LanHostState { }

class HostLobby extends LanHostState { }
class HostWaiting extends LanHostState { }
class HostCountdown extends LanHostState { }
class HostInGame extends LanHostState { }
class HostFinished extends LanHostState { }
class HostError extends LanHostState { }

// 任何 switch 表达式必须穷尽所有子类，否则编译报错
switch (state) {
  HostLobby()       => ...,
  HostInGame()      => ...,
  _                 => ...,  // 后面新增子类，这里必须加 case
}
```

### 状态机迁移图

Host 端：

```
HostLobby ──HostCreateRoomPressed──→ HostWaiting
HostWaiting ──HostClientJoined──→ HostWaiting(clientId/clientName 更新)
HostWaiting ──HostStartGamePressed──→ HostCountdown(3)
HostCountdown ──HostTick ×3──→ HostInGame(initialize(), room)
HostInGame ──HostMoveCommitted/Received──→ HostInGame(next)
HostInGame ──deviceLost──→ HostError
HostFinished ──HostStartGamePressed(再来一局)──→ HostCountdown(3)
```

Client 端对称。

**难度**：GamePage 的"再来一局"路径需要让 state 从 `HostFinished` 回到 `HostCountdown`（走倒计时）。但 reducer 原来的 `HostStartGamePressed when s is HostWaiting` 模式不匹配 `s is HostFinished` → **必须加 case**。这是 T6 的核心改动。

## 4. LocalNet 框架集成

### 4.1 框架三原语

`lib/core/localnet/` 框架提供三个核心原语供业务层使用：

| 原语 | API | 用途 |
|---|---|---|
| **设备发现** | `LanFramework.instance.watchDevices()` | 自动扫描局域网设备 |
| **通道消息** | `sendTo(peerId, channel, payload)` | HTTP 点对点定向通信 |
| **状态自动同步** | `Session<StateT extends Listenable>` | 监听 state 变化 → 自动序列化 → 发给对端 |

### 4.2 设备发现与房间协议

**设备发现**走 UDP 多播（`239.255.255.255:5678`，3s 心跳）：

```
Host 端 framework 周期发：deviceId,port[,extras]
Client 端 framework 收 → DeviceManager.onDatagram → 设备加入列表
```

**房间公告**（announceRoom）最初设计走 HTTP 定向 `sendTo(devices, payload)`。这有个致命问题——**对方设备还没被发现时，devices 列表为空，公告谁也收不到**。

**解决**：改为 UDP 多播（`sendMulticast`），不依赖 devices 列表。同时 `sendJoinRequest` / `sendJoinAccept`、`sendGameState` 也走同一多播通道：

```dart
// adapter 内 sendMulticast → LanFramework.sendMulticast → FrameworkCore
// → UdpTransport.sendRaw → RawDatagramSocket.send → 局域网全体
_fw.sendMulticast(key: 'room_announce', payload: {
  'toDeviceId': recipientDeviceId,  // 定向过滤
  ...innerPayload,
});
```

接收端统一在 adapter 的 `_multicastSub` 里按 `key` 和 `toDeviceId` 过滤。

### 4.3 Session 自动同步（核心难点）

#### 问题：Session channel 名不一致

`lib/core/localnet/session/session.dart` 内部 channel 名：

```dart
String get _sessionChannel => 'session/${peerId}_${state.hashCode}';
```

- Host 创建 Session：`peerId = clientDeviceId`，`state = Host.notifier`
- Client 创建 Session：`peerId = hostDeviceId`，`state = Client.notifier`

**peerId 不同 + hashCode 不同 → channel 名不同 → 双向不通！**

#### 解决：channelName 可选参数

给 `Session` / `SessionManager` / `LanFramework.createSession` 加可选 `String? channelName`：

```dart
Session({
  ...
  this.channelName,
}) : ...;

String get _sessionChannel =>
    channelName ?? 'session/${peerId}_${state.hashCode}';
```

双端都传 `channelName: LanChannels.gameState`（`'surround/game/state'`），channel 名一致，双向互通。

**向后兼容**：不传 `channelName` 时行为完全不变。

#### 解决：Session 异步发送覆盖问题

Session 内部用 `Timer(Duration.zero, ...)` 做微任务批处理：

```dart
void _onStateChanged() {
  if (_suppressSync) return;  // 循环抑制
  _syncTimer?.cancel();
  _syncTimer = Timer(Duration.zero, () async {
    final data = _serializer.serialize(state);
    await _channelManager.sendTo(peerId, _sessionChannel, data);
  });
}
```

**问题**：Host 端落子后 notifier → Session 异步序列化发 → 这个异步任务可能在 Client **回送新 state** 之后才执行 → Host 的 state 被旧值覆盖 → 棋盘回退。

**最终方案**：**不再使用 `Session<ValueNotifier<GameState>>` 的异步发送，改用双端同步 `sendGameState / watchGameState`（当前实现），留作未来升级。**

> **教训**：Session 抽象在状态同步场景不是银弹——它的异步发送 + 微任务批处理 + 循环抑制机制增加了时序复杂性。当前实现用**显式多播 + toDeviceId 过滤**替代 Session 的自动同步，虽然失去了"赋值即同步"的简洁性，但获得了**可预测的同步时序**。

## 5. 棋盘翻转与触摸镜像

### 5.1 问题

两设备对局，各自视角不同：
- Host 是 top player，但作为本地用户应该在**视觉下方**操作
- Client 是 bottom player，视觉也应在**下方**
- 棋盘必须**翻转**给用户

### 5.2 Plan D：Transform.flip + y 坐标镜像

```dart
// LanBoardStack(flipY: true)
// host 端棋盘用 Transform.flip 整个翻转
if (flipY)
  Transform.flip(
    flipY: true,
    child: _buildDrawingLayer(boardSize, distance),
  )

// 触摸层 OUTSIDE 翻转——不翻转手势，只翻转绘制
if (isMyTurn)
  TouchView(...)
```

但翻转后的棋盘触摸坐标也反了 → 用户点"左上角"会被识别为"右下角"。

#### TouchController 子类 + y 镜像

```dart
class LanHostTouchController extends TouchController {
  final double boardSize;

  Offset _mirror(Offset p) => Offset(p.dx, boardSize - p.dy);

  @override
  void handleTouchBegan(Offset localPosition, ...) {
    super.handleTouchBegan(_mirror(localPosition), ...);
  }
  // handleTouchMoved / handleTouchEnded 同样镜像
}
```

`boardSize = cellSize * 11`（棋盘宽高像素）。

### 5.3 为什么 Client 不翻转

```
Host 端：
  LanBoardStack(flipY: true)     ← 棋盘翻转，视觉上 top player 在底部
  TouchController = LanHostTouchController(boardSize: boardSize)  ← y 坐标镜像同步

Client 端：
  LanBoardStack(flipY: false)    ← 棋盘不翻转
  TouchController = TouchController()  ← 标准触摸
```

## 6. 双源状态机一致性（最 Bug 密集区）

### 6.1 问题

GamePage（游戏页）需要两个信息：
1. **当前棋盘状态**（GameState）— 驱动 UI 渲染
2. **当前状态机状态**（HostLobby/Waiting/Countdown/InGame/Finished）— 驱动页面布局

最初设计：**VM 持有状态机 + 棋盘 GameState**，Page 从 VM 读。

但上一轮 Session 升级把 VM 从 GamePage 里删了（VM 只在 LanRoomPage 用）。GamePage 改为**直接持有 `ValueNotifier<GameState>`**。

于是出现**两个 GameState 源**：
- **VM 的 `HostInGame.gameState`** — 只在 `dispatch(HostMoveCommitted)` 时同步更新
- **`_gameStateNotifier.value`** — 收到对端推送也更新

### 6.2 经典 bug：状态回退

```
A（Host）走子 → VM 更新 gameState + notifier 更新 → 对端同步
B（Client）走子 → B 的 notifier 更新 → sendGameState → Host 收到
                     ↓
Host 只更新 _gameStateNotifier（不更新 VM）
                  ↓
A 再走子 → vm.dispatch → reducer 用 VM 内**旧 stateA1** 算 next
           ↓
棋盘回退到 stateA1（丢弃了 B 的走子）
```

### 6.3 修法（TDD 驱动）

分两阶段：

**阶段 1**：当收到对端 game_state 时，同步更新 VM：

```dart
// Host 端收到 Client game_state → dispatch HostGameStatePushed
void _onGameStateSub(GameState gs) {
  _gameStateNotifier.value = gs;
  if (_viewModel.value is HostInGame) {
    _viewModel.dispatch(HostGameStatePushed(gs));  // ← 关键
  }
}
```

`HostGameStatePushed` 在 VM reducer 里匹配 `HostInGame => HostInGame(newGameState, room)`，让 VM 跟上。

Client 端对称（`ClientGameStatePushed`）。

**阶段 2**：升级为双端 Session（自动同步），不再需要手动 bridge：

Session 自动 serialize notifier → 对端 deserialize notifier → **双方 notifier 永远同一份值**，VM 完全退出 game 状态管理。Phase 2 时删除阶段 1 加的 `HostGameStatePushed` / `ClientGameStatePushed` 事件和相关 reducer case。

## 7. 触摸优先级冲突：ConfirmActions vs TouchView

### 7.1 问题

LanBoardStack 使用 Stack 包含了**两个**子节点：

```dart
Stack(
  children: [
    // 子节点 0：绘制层（含 ConfirmActions 按钮 ✓ ✘）
    _buildDrawingLayer(boardSize, distance),
    // 子节点 1：触摸层（Listener 覆盖全棋盘）
    if (isMyTurn)
      TouchView(
        onPointerDown: _onPointerDown, ...  // → touchController.handleTouchBegan
      ),
  ],
)
```

用户点 ✓ 按钮时：
1. **TouchView 的 Listener.onPointerDown 先触发**（覆盖全棋盘，包括按钮位置）
2. `handleTouchBegan` → `phase` 从 `confirming` **被覆盖**为 `beganMove`
3. 然后 ConfirmActions 的 `GestureDetector.onTap` 触发 → `_onConfirm`
4. `_onConfirm` 第一行 `if (toc.phase != TouchPhase.confirming) return;` → **早退**
5. **棋子不动，用户点 ✓ 无效**

### 7.2 修法

在 `handleTouchBegan` / `handleTouchMoved` / `handleTouchEnded` 三个方法开头加：

```dart
if (phase == TouchPhase.confirming) return;  // 确认阶段不抢触摸权
```

让 TouchView 在 `confirming` 阶段**不处理触摸事件**，让 ConfirmActions 的按钮专属 `GestureDetector` 独占触摸。

## 8. 持久化与 UX 完善

### 8.1 本机别名持久化

`PlayerProfileService`（SharedPreferences）：

- key: `lan_player_alias`，存字符串
- 首次进入 lobby → 弹 aliasDialog → 输名字 → 存盘
- 二次进入 lobby → 读盘 → 直接用（不弹 dialog）
- adapter.start() 内预加载：`loadAlias() → if null use 'Player'`

### 8.2 deviceId 持久化

`DeviceIdService`（SharedPreferences + UUID v4）：

- 首次启动生成 UUID 落盘
- 后续每次 `adapter.start()` → `DeviceIdService.load()` → 注入 `FrameworkConfig(deviceId: persistedId)`
- **为什么必要**：LanFramework 每次 start 生成新 UUID，对端会同时看到"老 B 离线"+"新 B 上线"两条记录

### 8.3 再来一局

Host 在 Victory 弹窗点「再来一局」→ dispatch `HostStartGamePressed` → VM 倒计时 → `_gameStateNotifier.value = QuoridorEngine.initialize()` → Session 自动发 → Client 端同步 → Victory 弹窗消失。

决策难点：**GamePage 没有 VM，怎么 dispatch 给 VM？**

解法：给 GamePage 加 `LanHostViewModel _viewModel`（**仅用于「再来一局」状态机**），不参与 build 渲染（build 只从 `_gameStateNotifier` 读）。VM 的倒计时完成后，`_onConfirm`（实际是 VC 直接设 notifier）触发棋盘重置 → Session 推新 state → Client 端一致。

## 10. 通信协议汇总

### 10.1 通道清单

| 通道名 | 方向 | 用途 | 传输 |
|---|---|---|---|
| `surround/room/announce` | Host → 广播 | 房间公告 | UDP 多播 |
| `surround/room/join` | 双向 | join 请求 + 响应 | UDP 多播 |
| `surround/game/state` | 双向 | GameState 全量同步 | Session / 多播 |

### 10.2 协议消息格式（sealed class）

```dart
sealed class LanRoomEvent { }

class HostRoomAnnounced  extends LanRoomEvent { room, hostDeviceId, hostAlias }
class HostRoomClosed      extends LanRoomEvent { roomId }
class ClientJoinRequested extends LanRoomEvent { clientDeviceId, clientAlias, roomId }
class ClientJoinResult    extends LanRoomEvent { roomId, clientDeviceId, accepted, reason? }
class HostClientLeft      extends LanRoomEvent { /* 内部掉线事件 */ }
class ClientDisconnectedProtocol extends LanRoomEvent { /* 内部掉线事件 */ }
```

所有消息序列化共同格式：
```json
{ "type": "HostRoomAnnounced", "room": {...}, "hostDeviceId": "...", "hostAlias": "..." }
```

接收端用 `LanRoomEvent.fromJson(json)` 按 `type` 字段分发。未知 `type` 抛 `FormatException`（防协议版本不兼容）。

### 10.3 toDeviceId 过滤

所有多播消息都带 `toDeviceId` 字段。接收端按 `toDeviceId == myDeviceId` 过滤「是否是我的消息」。这解决了**多播是全网广播**的问题——每条消息指定接收者，其他设备直接丢弃。
